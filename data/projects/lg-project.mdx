### 1. FDS 관리자 대시보드 개발
**8개 WAS 서버 실시간 모니터링 및 이상 탐지 시스템**

LG 유플러스의 FDS(Fraud Detection System) 관리를 위한 실시간 대시보드를 개발했습니다. 8개의 WAS 서버에서 발생하는 요청/응답 데이터와 이상 탐지 현황을 실시간으로 모니터링하고, 관리자가 직관적으로 파악할 수 있는 시각화 인터페이스를 구현했습니다.

![LG U+](/images/projects/lg-project/image1.png)

<div className="problem-solution">

#### 💡 핵심 문제상황

<div className="challenge-item">

**🔥 대용량 데이터 처리 시 Bad Gateway 502 에러**
- **문제**: GROUP_CONCAT과 복잡한 조인으로 인해 쿼리 실행 시간이 8초 이상 소요되어 웹 서버 타임아웃 발생
- **기술 해결**: MySQL EXPLAIN PLAN으로 실행 계획 분석 후 컬럼에 복합 인덱스 추가하고 Redis 캐싱 레이어 도입
- **결과**: 쿼리 실행 시간 **60% 단축**(8초→3.2초), Bad Gateway 에러 **100% 감소**(일 15건→0건)

</div>

<div className="challenge-item">

**📊 서버별 데이터 집계 쿼리 성능 문제**
- **문제**: 서버별 요청 성공률, 평균 응답시간 등을 실시간으로 계산할 때 DB 부하가 과도하게 발생
- **기술 해결**: CONCAT, FLOOR, LPAD 함수를 조합해 시간 단위별로 데이터를 사전 집계, MyBatis foreach로 배치 처리
- **결과**: 서버 상태 조회 응답시간 **50% 단축**, DB CPU 사용률 **30% 감소**

</div>

</div>

### 2. LG 개인정보 동기화 시스템 (PersonSync)
**이종 시스템 간 고객 데이터 실시간 동기화 및 무결성 관리**

LG 유플러스의 Idauth(본인확인), CAS(고객관리), APIM(API게이트웨이) 시스템 간 고객 정보 불일치를 해소하고, 개인정보 보호법을 준수하며 안전한 데이터 동기화를 수행하는 통합 관리 시스템을 개발했습니다.

![LG U+](/images/projects/lg-project/image2.png)

<div className="problem-solution">

#### 💡 핵심 문제상황

<div className="challenge-item">

**⚠️ 시스템 간 데이터 불일치로 인한 업무 장애**
- **문제**: Idauth와 CAS 시스템 간 고객 정보 동기화가 지연되어 본인 인증 실패가 빈번하게 발생(일 평균 500건)
- **기술 해결**: RESTful API를 설계해 두 시스템의 데이터를 실시간으로 비교하고, 불일치 발견 시 APIM을 통해 동기화
- **결과**: 데이터 불일치율 **100% 감소**(500건→0건)

</div>

<div className="challenge-item">

**🔐 외부 시스템 연동 안정성 강화 및 개인정보 보안 체계 구축**
- **문제**: APIM과 외부 시스템 연동 시 간헐적인 네트워크 오류로 동기화가 실패하는 문제가 발생했으며, 개인정보 보호법 준수를 위해 민감정보 암호화 및 접근 이력 추적 기능이 부족한 상황
- **기술 해결**:
  - Spring Retry를 활용하여 지수 백오프 방식의 재시도 로직을 구현
  - Circuit Breaker 패턴을 적용하여 연쇄 장애를 방지
  - AES-GCM 256bit 암호화를 적용해 민감정보를 보호하고, ActHistService를 통해 모든 개인정보 접근을 로깅하는 시스템을 구축
- **결과**:
  - 네트워크 오류 시 자동 복구 및 장애 확산 방지로 동기화 안정성 향상
  - 개인정보 보호법 완전 준수 체계 구축 및 감사 요구 시 접근 이력 100% 추적 가능

</div>

</div>

### 3. LG 법인폰 인증 시스템
**모바일 SMS 단축 URL 기반 다단계 인증 처리**

LG 법인폰 사용자가 SMS로 받은 단축 URL을 통해 안전하고 편리하게 인증을 완료할 수 있는 시스템을 개발했습니다. 임직원과 일반 사용자를 구분하여 차별화된 인증 경험을 제공하고, 인프라부터 애플리케이션까지, 아파치·톰캣 Web/WAS 분리 구조로 전체 시스템을 구축·운영했습니다.

  <img src="/images/projects/lg-project/image3.png" alt="LG U+ 시스템 화면 1" />
  <img src="/images/projects/lg-project/image4.png" alt="LG U+ 시스템 화면 2" />
</div>

<div className="problem-solution">

#### 💡 핵심 문제상황

<div className="challenge-item">

**⏰ 5분 타임아웃 관리와 URL 보안 검증**
- **문제**: SMS 단축 URL의 5분 제한 시간 관리가 복잡하고, 무단 접근이나 중복 사용을 방지해야 했음
- **기술 해결**: 당일 파티션 키를 활용해 데이터 조회 성능을 최적화하고, URL 키의 유효성, 타임아웃, 사용 횟수 검증 로직 구현
- **결과**: 부정 접근 차단 **100%**, URL 검증 처리 속도 **40% 향상**

</div>

<div className="challenge-item">

**🚀 고트래픽 환경에서 성능 및 안정성 문제**
- **문제**: 피크 시간대 집중 접속으로 응답 지연과 서버 부하 발생
- **기술 해결**: Apache와 Tomcat을 mod_jk로 연동해 WAS와 웹서버를 분리하고, 로드밸런싱으로 트래픽 분산
- **결과**: 평균 응답 시간 **44% 향상**(3.2초→1.8초), 피크 시간대에도 안정적 서비스 제공

</div>

### 4. SMS 발송 시스템 로드밸런싱
**144개 에이전트 효율적 관리 및 테스트 전략 수립**

본인확인 시스템과 연계된 144개의 SMS 발송 에이전트를 효율적으로 로드밸런싱하고, 다수의 대행사 환경에서 안정적으로 동작하는 시스템을 구성했습니다. 복잡한 테스트 환경의 제약사항을 극복하기 위한 창의적 해결책을 제시했습니다.

![LG U+](/images/projects/lg-project/image5.png)

<div className="problem-solution">

#### 💡 핵심 문제상황

<div className="challenge-item">

**🧪 현실적으로 불가능한 테스트 시나리오**
- **문제**: 24개 대행사 중 12번만 테스트 환경에서 개인키를 보유해 테스트 가능했고, 모든 조합 테스트를 위해서는 24번의 재기동이 필요
- **기술 해결**: 12번 대행사를 기준으로 1,2번 서버에 6개씩 인스턴스를 배치하는 집중 테스트 전략을 수립하고, Redis 기반 SMS 시뮬레이터를 개발해 다른 대행사 환경을 모사
- **결과**: 테스트 시간 **80% 절감**, 품질팀 승인을 받아 현실적 테스트 전략 확립

</div>

<div className="challenge-item">

**📋 품질팀 설득과 문서화 필요성**
- **문제**: 제한적 테스트 환경에 대해 품질팀의 우려가 있어 테스트 전략에 대한 승인을 받기 어려웠음
- **기술 해결**: 테스트 제약사항을 체계적으로 분석하고, 대안 전략의 타당성을 통계적 데이터와 함께 문서화하여 설득력 있는 제안서 작성
- **결과**: 품질팀 승인 획득, 향후 유사한 제약 상황에서 활용할 수 있는 테스트 방법론 확립

</div>

### 5. 본인확인 유큐브 상호대사 데이터 전송
**대용량 회원 정보 안전한 외부 시스템 연동**

회원 정보의 신뢰성을 검증하기 위해 외부 시스템과 APIM을 통해 대용량 데이터를 안전하고 효율적으로 교환하는 시스템을 개발했습니다. 보안성과 성능을 동시에 만족하는 데이터 전송 아키텍처를 구축하고 전체 인프라를 직접 운영했습니다.

![LG U+](/images/projects/lg-project/image6.png)

<div className="problem-solution">

#### 💡 핵심 문제상황

<div className="challenge-item">

**💾 대용량 데이터 전송 시 타임아웃 및 메모리 부족**
- **문제**: 일 평균 100만 건의 회원 데이터를 한 번에 전송하려니 응답 시간이 30초를 넘어 타임아웃이 발생하고, 서버 메모리 부족으로 OutOfMemory 에러 발생
- **기술 해결**: 데이터를 1,000건 단위로 청크 분할 후 비동기 병렬 처리하고, Tomcat의 스레드 풀 설정을 활용해 최대 10개의 워커 스레드가 동시에 전송 작업을 수행하도록 구성. 각 청크는 독립적인 요청 처리 태스크로 실행되어 메모리 사용량을 분산시키고, Tomcat의 커넥션·스레드 관리 기능을 통해 자원 사용 효율과 처리 속도를 최적화
- **결과**: 데이터 처리 시간 **20% 단축**(8시간→6.4시간), 메모리 사용량 **60% 감소**로 안정성 확보

</div>

<div className="challenge-item">

**🔄 전송 실패 시 데이터 복구 어려움**
- **문제**: 네트워크 불안정이나 외부 시스템 장애로 전송 중 실패할 경우, 어느 부분까지 성공했는지 파악이 어려워 재전송 범위를 결정하기 곤란
- **기술 해결**: 각 청크별로 전송 상태를 DB에 기록하고, 지수 백오프 방식의 재시도 로직을 구현하여 실패한 청크만 선별적으로 재전송
- **결과**: 부분 실패 시에도 전체 재전송 없이 효율적 복구 가능, 데이터 손실 **0건**
